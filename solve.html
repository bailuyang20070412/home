⚠️ Question 7 – Problems encountered during testing ⚠️

  • 🔹 Non‑numeric / empty inputs: prompt() returns string; parseFloat() gives NaN if user cancels, leaves blank, or types letters. This leads to invalid calculations unless validated.
  • 🔹 Violation of assumptions (a ≤ 0 or delta < 0): Even though lab assumes a>0 and Δ ≥ 0, random testing with negative a produces swapped parabola orientation or division by zero if a=0. Negative Δ gives NaN from Math.sqrt() → no real roots.
  • 🔹 Floating point precision: For certain values (e.g., a=1, b=1e15, c=1) results may show tiny rounding errors due to JavaScript's IEEE754 arithmetic (e.g., x1+x2 slightly off -b/a).
  • 🔹 parseFloat partial parsing: Input like "5.6abc" returns 5.6 silently, user might expect error but program continues — hidden inconsistency.
  • 🔹 User experience & repetitive testing: The built-in prompts block the UI. However, the "Solve" button allows multiple tests without page refresh, but each time three prompts appear — clicking outside or cancel leads to null handling.
  • 🔹 Large coefficients risk: Values extremely big cause delta overflow? Not typical, but b*b could exceed Number.MAX_SAFE_INTEGER losing precision.
  • 🔹 Delta = 0 yields double root: Works fine, but alert shows same x1/x2 (expected). No logical issue but user might be confused about identical roots.
💡 Note from testing: Our implementation includes validation (checks for NaN, a≤0, Δ<0) to avoid silent errors and illustrate real-world pitfalls. Without validation, problems would be more severe.
